The CLI installs as two names for the same binary: `dormice` and `dor`
(the one you actually type). It talks to a daemon over the
[HTTP API](/docs/http-api) and connects with exactly two environment
variables — missing one produces an error naming it:

```sh
export DORMICE_ENDPOINT=http://127.0.0.1:3676
export DORMICE_API_TOKEN=<token>   # on the daemon host: grep ^DORMICE_API_TOKEN /etc/dormice/env
```

Errors print as one line on stderr (no stack traces at a shell prompt)
and exit 1.

## dor sandbox ls

```sh
dor sandbox ls
```

Lists every sandbox with its lifecycle state, as a table. Listing is
observation — it never wakes anything. (Table output strips control
characters, so an ANSI escape hidden in a sandbox key cannot reach your
terminal.)

## dor sandbox exec

```sh
dor sandbox exec <externalId> '<command>' [-t <seconds>]
```

Runs a shell command in the sandbox behind the key, waking it first. The
command is one quoted argument, executed as `bash -c`. Its stdout and
stderr pass through to yours, and **its exit code becomes `dor`'s exit
code** — so `dor sandbox exec` in a script behaves like the command
itself.

`-t, --timeout <seconds>` kills the command after that long
([enforced inside the container](/docs/commands)). An unknown key is a
404, never a sandbox created by accident.

## dor sandbox push / pull

```sh
dor sandbox push <externalId> <localPath> [remotePath]
dor sandbox pull <externalId> <remotePath> [localPath]
```

One file in, one file out. Relative remote paths land under
`/home/user`; `push` without a remote path uses the local file's name.
`pull` without a local path writes the raw bytes to stdout — pipe it
wherever. Both obey the [16 MiB single-file limit](/docs/files).

## dor sandbox rebuild

```sh
dor sandbox rebuild <externalId>
```

Swaps the container and keeps `/home/user` — the sandbox's next use
starts on its template's *current* image (or the base image). This is
how a [template upgrade](/docs/templates) reaches an existing sandbox.
Details: [Rebuild a sandbox](/docs/persistence#rebuild-a-sandbox).

## dor sandbox destroy

```sh
dor sandbox destroy <externalId>
```

Destroys the sandbox — container, disk, and (if archived) the S3
object. The only command that loses data. Idempotent: destroying a key
that has no sandbox reports so and exits 0.

## dor template

```sh
dor template add <name> <image>   # register, or re-point — that is the upgrade
dor template ls
dor template rm <name>
```

A template is a name for a Docker image already on the daemon's host.
`add` on an existing name re-points it at the new image — the
[upgrade front door](/docs/templates). `rm` is refused (with the
sandboxes named) while any sandbox still uses the template, and never
removes the image itself.

## dor doctor

```sh
dor doctor           # every check, including probes that start a real container
dor doctor --quick   # skip the container probes
```

Read-only host checks: can this machine run the daemon? Exits 0 unless
a check fails. Every check is documented one by one in
[dor doctor](/docs/doctor).
