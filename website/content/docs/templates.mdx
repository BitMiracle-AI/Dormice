---
title: Templates
description: Learn how to create sandboxes from your own Docker images by registering them as templates.
---

A template is a name pointing at a Docker image. Dormice deliberately
has no build pipeline — the host's Docker daemon is already an image
store, and `docker build` is already the build tool.

## Create a template

Build an image (any way you like) and register its name:

```sh
docker build -t my-agent-env:v1 .
dor template add my-agent-env my-agent-env:v1
dor template ls
```

## Create sandboxes from a template

Both API surfaces accept a template name at creation:

```ts
// Native: a creation-time option, like policy
await client.acquireSandbox('user-42', { template: 'my-agent-env' });

// E2B: the templateID argument
const sbx = await Sandbox.create('my-agent-env', { /* the two URLs + apiKey */ });
```

An unknown template name is refused — 400 natively, 404 through the E2B
surface — never silently substituted with the base image.

## Update a template

Re-registering a name points it at a new image. Sandboxes pick up the
new image the next time their container is built — on a rebuild, a cold
start, or a restore from archive:

```sh
docker build -t my-agent-env:v2 .
dor template add my-agent-env my-agent-env:v2  # re-registering re-points the name
dor sandbox rebuild user-42                    # per sandbox: new container, same /home/user
```

Until rebuilt (or cold-started), existing sandboxes keep running the
image they were born on. Through a rebuild, `/home/user` carries over
untouched and everything outside it resets onto the new image — see
[Rebuild a sandbox](/docs/persistence#rebuild-a-sandbox).

You can ask which sandboxes still run an old image with the
`listSandboxImages` verb (SDK method of the same name), or look for the
"upgradable" marker in the [console](/docs/console)'s sandbox list.

**Note:** the comparison is by image name. Rebuilding the same tag in
place is invisible to it — upgrade by registering a new tag
(`my-agent-env:v2`), not by overwriting `v1`.

## Remove a template

```sh
dor template rm my-agent-env
```

Removing a template that sandboxes still use is refused with a 409
naming a sandbox. Removing an unknown name is a no-op, and removing a
registration never removes the image itself.

## Image requirements

Sandboxes mount their own disk over `/home/user` (see
[Anatomy of a sandbox](/docs/architecture#anatomy-of-a-sandbox)), so a
template must
install its tools into image paths (`/opt`, `/usr/local`) — anything
baked under `/home/user` is hidden at runtime. The stock image's
[`Dockerfile`](https://github.com/BitMiracle-AI/Dormice/blob/main/images/Dockerfile)
is the reference: layers ordered by change frequency, tools outside the
home directory, non-root user at uid 1000.

## Good to know

- **Registration is configuration.** The image is not checked for
  existence when you register — register first, build later, if you
  like. If it is still missing when a sandbox needs it, creation fails
  with a named error
  (`image X is not on this host — docker pull or build it, then retry`).
- **`base` is reserved.** Through the E2B surface, `Sandbox.create()`
  with no template argument means the daemon's base image; the name
  `base` cannot be registered.
- **Templates apply at creation only.** An acquire with a `template` on
  a key that already has a sandbox keeps the sandbox it finds, template
  unchanged — but an unknown name is still refused, never silently
  ignored.
