title: Moving files
description: Learn how to write and read files in a sandbox, from the SDK or the CLI — and what to do with files over the 16 MiB limit.

## Write files

You can write files into a sandbox with `writeFile` (one file) or
`writeFiles` (a batch). Both wake a cold sandbox first; paths are
absolute, or relative to `/home/user`:

```ts
await client.writeFiles('my-agent', [
  { path: 'config.json', content: JSON.stringify(config) }, // string → UTF-8
  { path: 'data/input.bin', content: bytes },               // Uint8Array, as-is
]);

await client.writeFile('my-agent', 'notes.md', '# one file, no array');
```

Parent directories are created as needed; existing files are
overwritten.

## Read files

`readFile` reads one file; `readFiles` reads a batch, returned in
request order:

```ts
const file = await client.readFile('my-agent', 'results/out.csv');
const text = new TextDecoder().decode(file.content);

const files = await client.readFiles('my-agent', ['a.txt', 'b.txt']);
```

A missing file is a 404; a directory or other non-regular file is a 400.

## Move files with the CLI

`dor sandbox push` and `pull` move single files:

```sh
dor sandbox push my-agent ./data.csv            # → /home/user/data.csv
dor sandbox push my-agent ./data.csv inputs/d.csv
dor sandbox pull my-agent results/out.csv out.csv
dor sandbox pull my-agent results/out.csv       # no local path: raw bytes to stdout
```

## Size limits: 16 MiB per file

File content crosses the wire as base64 inside JSON, so each file is
capped at 16 MiB, write and read alike. A `writeFiles` batch tops out at
48 MiB of request body, and a `readFiles` batch at 48 MiB total.

Reading a file over the cap returns a **413 naming the file's actual
size** — never a truncated file, which would just be a corrupt file.

For bigger files, use one of these instead:

- **Download from inside the sandbox.** Run `curl` or `wget` (both in
  the stock image) with [`execCommand`](/docs/commands) — full speed, no
  JSON layer in the path.
- **Use the E2B files surface.** `files.read` / `files.write` stream
  with no artificial cap — the disk quota is the cap — and
  `uploadUrl()` / `downloadUrl()` mint signed URLs that a bare browser
  or curl can use, no headers required. See
  [Using the official E2B SDKs](/docs/e2b-sdks).

## Batch behavior

- **`writeFiles` fails fast, not atomically.** Files are written in
  array order; on an error, the earlier files are already written.
  Splitting a big upload into several calls loses nothing — writes are
  independent per file.
- **`readFiles` is all or nothing.** One missing path fails the whole
  call with a 404 naming it — a batch read that silently dropped a file
  would hand you a corrupt project checkout. If you can tolerate missing
  files, read them one by one.

## How file transfer works

File operations execute *inside* the sandbox, streamed through an
in-container process — never as host-side writes into the disk image.
This means a hostile symlink planted in the sandbox can only point at
the sandbox's own filesystem, and ownership lands on the sandbox user
naturally. See [Architecture](/docs/architecture) for the boundary this
preserves.
