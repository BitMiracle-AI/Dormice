---
title: Metrics
description: Learn how to read live metrics for a sandbox and for the host — without ever waking anything.
---

Dormice answers "how are things *right now*" with single snapshots —
deliberately not a monitoring system. For history, trends, and alerting,
point a real monitoring stack at the box.

One rule governs all metrics: **observation never wakes a sandbox and
never resets its idle clock** — the same principle as listing
([what counts as activity](/docs/lifecycle#what-counts-as-activity)).
You can watch a fleet of frozen sandboxes from a dashboard and they stay
exactly as cheap as if you ignored them.

## Get metrics for one sandbox

The native `getSandboxMetrics` verb returns one live sample — CPU count
and usage, memory, and disk:

```ts
const sample = await client.getSandboxMetrics('my-agent');
if (sample) {
  sample.cpuUsedPct;    // percent of one CPU; can exceed 100
  sample.memUsedBytes;
  sample.memTotalBytes;
  sample.diskUsedBytes; // real occupancy of the sandbox's disk
  sample.diskTotalBytes;
}
```

The E2B SDK's `getMetrics()` reads the same numbers, returned as an
array with one sample. Either way:

- A **frozen** sandbox is readable without waking it.
- A **stopped** (or archived, or restoring) sandbox has no running
  container, so the answer is honestly empty — `null` natively, an
  empty array through E2B — never zeros pretending to be measurements.
- The E2B SDK's `start` / `end` range parameters are accepted and
  ignored — there is no metrics history, only now.

## Get metrics for every sandbox at once

`listSandboxMetrics` answers for the whole fleet in a single request —
a dashboard over N sandboxes costs one call, not N:

```ts
const samples = await client.listSandboxMetrics();
// [{ externalId, sandboxId, sample }, …]
```

Only sandboxes with a running or frozen container appear; colder
sandboxes are simply absent — the same honesty as `null`, expressed as
absence.

## Get metrics for the host

The native `getHostMetrics()` answers the operator's question — is this
machine okay, and what do the sandboxes collectively cost it?

```ts
const m = await client.getHostMetrics();
```

The snapshot has four sections:

| Section | What it tells you |
| --- | --- |
| `host` | CPU usage, available memory, and **swap** — the freeze mechanism's fuel. Frozen sandboxes live in swap; a full swap means "idle is free" is ending. |
| `dataDisk` | The filesystem holding sandbox disks (`DORMICE_DATA_DIR`). This is the real capacity ceiling: past full, even the daemon's database cannot write. |
| `sandboxes` | Totals from the daemon's records: sandbox count, the `DORMICE_MAX_SANDBOXES` cap, and a count per lifecycle state. |
| `sandboxDisks` | What the sparse disk images *promise* (nominal) versus what they *occupy* (actual). The gap is your disk overcommit — watch it, because nothing else caps it. Measured on the test host: a sandbox promising 10 GiB actually occupied 68 MiB. |

**Readings are honest about their limits:**

- CPU usage is computed as the delta between your calls, so the first
  call after a daemon start answers `null` rather than a made-up zero.
- Memory is reported as `MemAvailable` — what could still be allocated
  without swapping. (The naive "free" number is misleadingly low on any
  healthy Linux box.)
- `swap.totalBytes: 0` is a real reading of a machine with no swap;
  `null` means the platform offers no reading at all.

The web console's Overview page renders exactly this snapshot — CPU,
memory, swap, the data disk, sandbox states, and the overcommit meter —
refreshed every few seconds. Readings a platform cannot produce show as
an amber hint pointing at `dor doctor`, never a fake number.
