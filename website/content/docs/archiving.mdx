---
title: Archiving to S3
description: Learn how to move long-idle sandboxes off the host into any S3-compatible bucket, and restore them with real progress reporting.
---

Frozen costs a few MiB of RAM; stopped costs only disk. Archiving takes
the last step: the sandbox's disk is packed into one compressed file
(with `tar` + `zstd`) and shipped to an [S3-compatible
bucket](/docs/core-concepts#object-storage-s3) — rented internet
storage — and the host keeps **nothing**. A machine with a modest disk
can hold a long tail of dormant sandboxes for the price of object
storage.

## Enable archiving

Set the four `DORMICE_S3_*` variables
([reference](/docs/configuration#archiver-s3)). Any S3-compatible store
works: AWS, Cloudflare R2, MinIO, Alibaba OSS in S3-compat mode.

With them set, stopped sandboxes archive after 7 more idle days by
default. Tune it per sandbox with the policy:

```ts
await client.acquireSandbox('user-42', {
  policy: {
    stopAfterSeconds: 3600,         // an hour idle → stopped
    archiveAfterSeconds: 7 * 86400, // a week idle → the bucket
  },
});
```

Without the S3 variables, archiving is off: sandboxes park at stopped
forever, and a policy asking to archive is rejected at acquire time —
never stored as a promise the daemon cannot keep.

## Restore an archived sandbox

`acquireSandbox` on an archived key starts the restore and answers
immediately with real progress — never a silent multi-minute hang:

```ts
let res = await client.acquireSandbox('user-42');
while (res.status === 'restoring') {
  console.log(res.progress.phase, `${res.progress.percent}%`); // downloading | extracting
  await new Promise((r) => setTimeout(r, 1000));
  res = await client.acquireSandbox('user-42');
}
```

Progress is measured, not invented: download percent against the
object's actual size, extraction percent against bytes fed to `tar`.
While a restore runs, command and file verbs answer 409 pointing back at
acquire, and destroy is refused too — retry when the restore finishes.

The E2B surface has no `restoring` concept, so there `connect` /
`create` simply block until the sandbox is ready — faithful E2B
semantics.

The whole loop is verified end to end against MinIO: a sandbox idles
down to archived, the bucket object appears and the local disk is freed;
re-acquiring reports `restoring`, flips to `ready`, and the files come
back byte-identical.

## The trade-off

Archiving converts disk money into wake-up time: a stopped sandbox
cold-starts in seconds, an archived one needs a download and an unpack
first — proportional to how much the sandbox actually wrote. Raise
`archiveAfterSeconds` to keep sandboxes on disk longer, or set it to
`null` for sandboxes that must never wait.

As a bonus, a restore builds its disk at the *current* size setting and
boots the *current* image —
[restores upgrade too](/docs/persistence#restores-upgrade-too).

## How archiving stays crash-safe

The archiver holds one invariant: **the local disk is deleted only after
the upload is confirmed and recorded.** A crash anywhere in between
leaves either a stopped sandbox (the retry re-archives to the same
object key) or an archived one — never a sandbox whose only copy is a
half-written object. A failed restore deletes its half-built disk and
returns the sandbox to archived; the object was never touched, so the
retry is clean.

There is one object per sandbox (`disks/<sandboxId>.tar.zst`). A
successful restore deletes the object, and destroying an archived
sandbox deletes it too — nothing leaks in the bucket. If you unset the
S3 variables while sandboxes sit archived, touching one answers a 503
naming the missing variables.
