The daemon hosts a web console at `http://127.0.0.1:3676/console` — same
process, same port, nothing extra to run. Everything it shows comes from
the same API documented in this section, and it follows the same rule:
**watching never wakes a sandbox; using one does, and only behind an
explicit click.**

(The console UI is currently Chinese-language.)

## Signing in

The console has one account with a username and password. On first
visit it walks you through creating it — you prove you are the operator
by pasting the API token once, then choose the credentials you will
actually sign in with. Forgot the password? Present the token again and
set a new one; repeated wrong guesses are slowed down with increasing
delays.

Signing in sets an `httpOnly` session cookie (valid 7 days) — a cookie
page JavaScript cannot read, so a script injected into the console
cannot steal your credential. The API token and the console password
rotate independently: changing the password does not break SDK clients,
and rotating the token does not sign browsers out. Resetting the
account (via the token) invalidates every existing session.

## Overview

The "is this machine healthy" page: CPU, memory, **swap** and the **data
disk** as first-class cards — swap filling up means "idle is free" is
ending, and a full data disk is the real capacity ceiling. Below them:
sandbox counts by lifecycle state and the disk-overcommit figure
(promised vs actually occupied — the same numbers as
[`getHostMetrics`](/docs/metrics)). Usage bars turn amber at 75% and red
at 90%; a host without swap gets an amber warning pointing at
`dor doctor`.

## Sandboxes

The live list — every sandbox with its state and live CPU, memory, and
disk columns, refreshed every few seconds, with search, a state filter,
sorting, and multi-select batch destroy. Creating a sandbox from here
is exactly `acquireSandbox`: a name, the lifecycle knobs (including the
never-stop [resident-agent switch](/docs/resident-agents)), and a
template dropdown when templates are registered. After a
[template upgrade](/docs/templates#update-a-template), sandboxes still
running the old image carry an "upgradable" marker next to their
template name — rebuild them to move.

A sandbox's detail page is a workspace:

- **Overview** — identity, policy thresholds with a countdown to the
  next lifecycle step, last activity; an adjust-policy dialog (the
  console's face of
  [`updatePolicy`](/docs/lifecycle#change-the-policy-later)), plus
  destroy and [rebuild](/docs/persistence#rebuild-a-sandbox) buttons,
  each behind a confirmation dialog.
- **Terminal** — a real interactive shell, kept behind an explicit
  "open" button because opening one *wakes* the sandbox. The browser
  speaks the same wire the official E2B SDK does. Closing the terminal
  kills its shell rather than leaving an orphan behind.
- **Files** — browse, preview, edit, upload (drag and drop works),
  download, rename, delete under `/home/user`, refreshing live while
  the sandbox runs. File operations wake the sandbox too, so a cold
  sandbox shows an explicit "open" gate first.
- **Processes** — the E2B process table, polled without waking (a
  stopped sandbox honestly shows none).
- **Metrics** — the sandbox's live resource snapshot, with an
  in-session trend chart (the browser accumulates it — the daemon
  [keeps no history](/docs/metrics)).
- **History** — this sandbox's recent lifecycle events.

## Templates

Register a name for an image, re-register to re-point it — the console's
form is the same [upgrade front door](/docs/templates) as
`dor template add`. Deleting a template a sandbox still uses is refused
with the daemon's own message naming the sandboxes.

## API keys

Mint revocable credentials without leaving the browser
([what they are](/docs/http-api#api-keys)). The create dialog takes a
name and an optional expiry date, then shows the key **exactly once** —
the daemon stores only a hash — with a copy button, a "copy connection
config" button (endpoint + credential, ready for CI secrets), and no
second chance. The table keeps every key ever minted, revoked ones
included, as your rotation history: name, masked prefix, creation time,
last use, expiry, an enable/disable switch (the reversible half of
revoking), and status. Rows can be edited in place (rename, change or
clear the expiry), selected in bulk for revocation, and revoking takes
effect on the key's next request. `DORMICE_API_TOKEN` itself appears as
a pinned, read-only "default" row so all credentials are visible in one
table — but it lives in the server's environment, so retiring it means
editing `/etc/dormice/env`, and it (or a console session) is also the
only credential the key-management actions themselves accept: keys
cannot manage keys.

## Activity, domains, and settings

Three more pages round out the operator view:

- **Activity** — the daemon's recent history (who was created, frozen,
  destroyed, what the reconciler repaired, which API key was minted or
  revoked), each event attributed to the credential that asked — an API
  key by name, the env token, a console session, or "system" for the
  daemon's own moves. Filterable by event kind, actor, and sandbox
  name: filtering by a key is how you read a leaked key's blast radius
  before revoking it. A bounded window, not an audit log.
- **Domains** — bind domains to the daemon's front door (the managed
  reverse proxy), with a copyable DNS record guide and a live per-domain
  probe that shows DNS and certificate progress until each one turns
  green. See
  [Installation](/docs/installation#connect-from-another-machine).
- **Settings** — the daemon's effective configuration, read-only, with
  secrets shown as present-or-absent only. Changing configuration
  happens in `/etc/dormice/env` plus a restart, never from the browser.

A command palette (⌘K) jumps to any page or sandbox.

## Connect

Copy-paste snippets for the SDK, the official E2B packages, and curl,
pre-filled with the endpoint the browser itself is using. The API token
is never displayed — the console does not hold it and cannot read it
back; fetch it from the daemon host
(`grep ^DORMICE_API_TOKEN /etc/dormice/env`).

## What you will not find

Development builds show extra preview pages backed by sample data,
clearly badged; a production build hides them entirely. The rule: a real
deployment must never render a fake all-green health report.
