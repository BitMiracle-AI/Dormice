title: What is Dormice?
description: Dormice is a self-hosted sandbox platform for AI agents — run it on your own machine and give every agent a secure, permanent sandbox.
---

Dormice gives your AI agents isolated Linux sandboxes where they can
safely run code, process data, and keep a long-lived workspace — on a
machine you already own. You install one daemon (a program that runs in
the background continuously); it manages the sandboxes for you.

**New to Docker or self-hosting?** These docs explain every term the
first time it matters, and [Core concepts](/docs/core-concepts) collects
them all in one place — start there if words like container, image, or
reverse proxy are new to you.

Start a sandbox and run a command in a few lines:

```ts
import { Dormice } from '@dormice/sdk';

const client = new Dormice({
  endpoint: 'http://127.0.0.1:3676',
  token: process.env.DORMICE_API_TOKEN!,
});

await client.acquireSandbox('my-agent');
const result = await client.execCommand('my-agent', 'echo "Hello from Dormice!"');
console.log(result.stdout); // Hello from Dormice!
```

## Why Dormice

- **Sandboxes are permanent.** Cloud sandbox platforms charge for every
  second a sandbox exists, so their sandboxes are disposable. Dormice
  inverts that: `acquireSandbox('my-agent')` always returns the same
  sandbox for the same key — today, tomorrow, next month — with all its
  files intact.
- **Idle costs almost nothing.** Idle sandboxes cool down on their own
  (`active → frozen → stopped → archived` — the
  [lifecycle](/docs/lifecycle)), and any acquire brings them back.
  Measured on real hardware: freezing takes an idle sandbox holding
  1 GiB of memory down to ~5 MiB, and waking it takes ~50 ms.
- **Deploys like a single binary.** One daemon, one SQLite database (a
  database that lives in a single file), one port. No Kubernetes, no
  external database.
- **Works with the official E2B SDKs.** [E2B](https://e2b.dev) is a
  popular commercial sandbox cloud that many agent frameworks build on;
  its unmodified `e2b` packages (JavaScript and Python) run against
  Dormice by changing two URLs — see
  [Using the official E2B SDKs](/docs/e2b-sdks).

## How it works

The daemon runs each sandbox as a Docker
[container](/docs/core-concepts#containers-and-images) hardened with
[gVisor](/docs/core-concepts#gvisor) (an extra isolation layer that
keeps untrusted code away from your real machine), and gives each one a
disk of its own. When a sandbox sits idle, the daemon suspends its
processes and squeezes their memory out to
[swap](/docs/core-concepts#swap-and-freezing) — disk space the
operating system uses as overflow memory — so it costs a few megabytes
of RAM and wakes in about 50 ms, processes still running. The sandbox's
data lives on its own disk, not in the container, so it survives daemon
restarts, host reboots, and even the container being deleted.
[Architecture](/docs/architecture) has the full picture.

## Status

Early development. The daemon, SDK, CLI, web console, Docker + gVisor
executor, and E2B-compatible API work end to end and pass a black-box
test suite against real infrastructure. Nothing here is ready for
production yet.
