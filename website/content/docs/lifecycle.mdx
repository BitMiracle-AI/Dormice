---
title: Sandbox lifecycle
description: Idle sandboxes cool down through four states to save resources, and any acquire brings them back. Learn how to tune when that happens.
---

A sandbox is always in one of four states. The longer it sits idle, the
colder — and cheaper — it gets, one step at a time. `acquireSandbox`
brings it back from any state.

```text
active ── idle ──▶ frozen ── idle ──▶ stopped ── idle ──▶ archived
  ~full RAM         ~5 MiB RAM         disk only            S3 only
                    wakes in ~50 ms    cold start (seconds)  restore (download)
```

## The four states

- **active** — the container is running, normally.
- **frozen** — every process is suspended and its memory pushed out to
  swap. Measured on real hardware: an idle sandbox holding 1 GiB drops to
  about 5 MiB of resident memory, and wakes in about 50 ms with all its
  processes still running, exactly where they were.
- **stopped** — the container is shut down; only the disk remains.
  Waking is a cold start from the surviving disk — seconds rather than
  milliseconds. Processes do not survive this step; files do.
- **archived** — the disk is compressed and shipped to an S3-compatible
  bucket; nothing remains on the host. This state only exists when you
  [configure archiving](/docs/archiving). Waking downloads and unpacks
  the disk, with real progress reporting.

## Set the lifecycle policy

Each sandbox carries three thresholds, all counting seconds since its
last activity. Pass them when you create the sandbox:

```ts
await client.acquireSandbox('my-agent', {
  policy: {
    freezeAfterSeconds: 60,      // freeze after 1 minute idle
    stopAfterSeconds: 7 * 86400, // stop after a week idle
  },
});
```

| Knob | Default | `null` means |
| --- | --- | --- |
| `freezeAfterSeconds` | 10 minutes | — (freezing is always on) |
| `stopAfterSeconds` | 3 days | never stop |
| `archiveAfterSeconds` | 7 days when S3 is configured, otherwise never | never archive |

Because all three count on the same clock, they must be ordered:
freeze ≤ stop ≤ archive.

**Good to know:**

- **Policy is fixed at creation.** A policy sent while acquiring an
  *existing* sandbox is not applied — the sandbox keeps the policy it was
  born with. (An invalid policy is still rejected with a 400.)
- **Archiving requires stopping.** Only a stopped sandbox can archive,
  so `archiveAfterSeconds` needs a non-null `stopAfterSeconds`. On a
  daemon without S3 configured, a policy asking to archive is rejected
  outright rather than silently stored.
- **Your explicit values win over defaults.** If you push
  `stopAfterSeconds` past the 7-day archive default, the archive
  threshold follows it instead of turning your valid setting into an
  ordering error.

A background scanner checks every `DORMICE_SCAN_INTERVAL_SECONDS`
(default 60) and moves each idle sandbox down at most one state per
sweep.

## Keep a sandbox warm forever

For a long-lived agent's sandbox, set `stopAfterSeconds: null`:

```ts
await client.acquireSandbox('my-agent', { policy: { stopAfterSeconds: null } });
```

The sandbox parks at **frozen** when idle — a few MiB of RAM, ~50 ms to
wake — and never decays to a cold start. Its processes are never killed
by idleness; they suspend and resume mid-flight, indefinitely. See
[Resident agents](/docs/resident-agents) for the full pattern.

## What counts as activity

Running commands, file operations, and `acquireSandbox` reset the idle
clock. A long-running command keeps its sandbox alive for as long as it
runs.

Observation deliberately does not count: `listSandboxes`, sandbox
metrics, and the console's list and detail views never wake a frozen
sandbox and never reset its clock. You can watch a whole fleet from a
dashboard without keeping anything warm.

## E2B timeouts are a separate clock

Sandboxes created through the [E2B surface](/docs/e2b-sdks) additionally
carry an absolute deadline (`timeoutMs`), because E2B semantics require
one: at the deadline the sandbox is killed — or parked, with
`lifecycle: { onTimeout: 'pause' }`. Activity does not extend a deadline;
only `create`, `connect`, and `setTimeout()` do. The deadline rule runs
before the idle rules above.

Natively created sandboxes never get a deadline — permanence is the
native default, and the E2B compatibility layer never imposes deadlines
on sandboxes it does not own.
