---
title: Running commands
description: Learn how to run shell commands inside a sandbox — timeouts, working directory, environment variables, and output limits.
---

Running commands is how your code (or your agent) actually does things
inside a sandbox — install packages, run scripts, execute whatever the
agent wrote. This page covers the native `execCommand` verb end to end.

## Run a command

You can run a shell command inside a sandbox with `execCommand`. It
wakes the sandbox if needed, runs the command as `bash -c` (as the
sandbox's non-root user), waits, and returns the whole result at once:

```ts
const result = await client.execCommand('my-agent', 'python3 --version');
result.exitCode;        // 0
result.stdout;          // 'Python 3.12.…\n'
result.stderr;          // ''
result.stdoutTruncated; // false — see "Output limits" below
```

**A non-zero exit code is a result, not an error.** `execCommand`
resolves with `exitCode: 1` — it does not throw. Errors are reserved for
the protocol failing: 404 when no sandbox exists behind the name (one is
never silently created), 409 while a sandbox is mid-restore.

## Run a command from the CLI

`dor sandbox exec` runs the same verb from a shell. The command's stdout
and stderr pass through to yours, and its exit code becomes `dor`'s own —
so it behaves like the command itself in scripts:

```sh
dor sandbox exec my-agent 'uname -r'
dor sandbox exec my-agent 'npm ci && npm test' -t 3600
```

## Set a timeout

Pass `timeoutSeconds` (default 300, maximum 86400). On expiry the
command is forcibly killed and returns exit code 137 — the Linux
convention for a process that was killed rather than allowed to finish:

```ts
await client.execCommand('my-agent', 'npm ci && npm test', {
  timeoutSeconds: 3600,
});
```

The SDK derives its HTTP deadline from the same value, so a long-running
command is never cut short by a client-side timeout.

**Note:** there is no `timedOut` flag in the response — 137 is also what
an out-of-memory kill inside the sandbox looks like, and the daemon
cannot tell the two apart, so it does not pretend to.

## Set the working directory and environment

```ts
await client.execCommand('my-agent', 'env | sort', {
  cwd: '/home/user/project',
  env: { NODE_ENV: 'test' },
});
```

`cwd` defaults to `/home/user` — the same base that relative file paths
resolve against.

## Output limits

stdout and stderr are each capped at 1 MiB. Truncation is always
reported (`stdoutTruncated` / `stderrTruncated`), never silent. For bulk
output, write to a file and [read it out](/docs/files) instead of
routing everything through stdout.

## A running command keeps the sandbox awake

While a command runs, the sandbox's idle clock keeps resetting — the
lifecycle engine never freezes a sandbox mid-command, even one that
legitimately runs for hours.

## Streaming, background processes, and terminals

`execCommand` is deliberately simple: one request, one finished result.
For live streaming output, processes that outlive the request, stdin, or
an interactive terminal, use the [E2B surface](/docs/e2b-sdks) —
`commands.run` streams, backgrounds, and reconnects, and `pty.*` is a
real interactive bash.

## How timeouts are enforced

The timeout runs *inside* the container (`timeout --signal=KILL` around
your command), not as a host-side deadline. Dropping the HTTP request
cannot reach a process on the other side of the container boundary — an
in-container timeout actually kills the command instead of just
reporting that it should die.
