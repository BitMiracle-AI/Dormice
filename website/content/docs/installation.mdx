## Install the daemon

Run this as root on a bare Ubuntu/Debian x86_64 host:

```sh
curl -fsSL https://raw.githubusercontent.com/BitMiracle-AI/Dormice/main/deploy/install.sh | bash
```

If your host has a slow connection to GitHub and the usual package
sources, add `-s -- --mirror cn`.

The installer sets up everything — Docker, gVisor, swap, the network
hardening, the daemon as a systemd service — and finishes by running
`dor doctor`, a battery of read-only checks that verifies the install
actually works. You can re-run `dor doctor` at any time.

**The installer is safe to re-run.** Running it again upgrades the code
and repairs anything that drifted, and never rotates your API token.

## Get your API token

The installer generates an API token and stores it in the daemon's
config file. You need it for every client — SDK, CLI, and console:

```sh
grep ^DORMICE_API_TOKEN /etc/dormice/env
```

## Connect from another machine

The daemon listens on `127.0.0.1` only — reaching it from another
machine is a choice you make explicitly, one of two ways.

**Option 1: SSH tunnel** (private, zero setup):

```sh
ssh -L 3676:127.0.0.1:3676 root@your-host
```

Then use `http://127.0.0.1:3676` as your endpoint, as if the daemon were
local.

**Option 2: Reverse proxy** — serves the API, the E2B surface, and the
web console at once. With Caddy, TLS certificates are automatic once you
give it a domain:

```text
your-domain.example {
	reverse_proxy 127.0.0.1:3676 {
		flush_interval -1
	}
}
```

**Note:** `flush_interval -1` matters — streamed command output is
written frame by frame, and a buffering proxy would hold it all back
until the end. Use HTTPS for anything exposed beyond localhost: the API
token and the session cookie travel in every request.

## What the host needs

The installer handles all of this and `dor doctor` verifies it, but
these are the requirements underneath:

- **Docker + gVisor (`runsc`)**, and root — sandboxes are gVisor
  containers, and the daemon needs loop mounts and cgroup writes.
- **Swap, with `vm.swappiness=100`.** Freezing works by pushing idle
  sandbox memory to swap. gVisor holds sandbox memory in a form the
  kernel refuses to swap at the default setting — measured on real
  hardware: 0 bytes reclaimed at the default, 99.5% reclaimed at 100.
  Some cloud images (Alibaba Cloud, for one) ship `vm.swappiness=0`;
  check the effective value with `sysctl vm.swappiness`.
- **Network hardening.** Sandboxes run untrusted code, so the installer
  blocks the cloud metadata service from containers
  (`iptables -I DOCKER-USER -d 169.254.0.0/16 -j DROP`, persisted) and
  disables inter-container traffic (`"icc": false` in `daemon.json`).
- **One machine, one daemon.** The daemon enforces this with a lock and
  refuses to start when its database and the machine's reality don't
  match.

Without a prepared Linux host, the daemon still runs anywhere Node 22+
runs using an in-memory fake executor — useful for development, not for
real sandboxes.
