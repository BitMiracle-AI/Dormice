title: Sandbox persistence
description: Everything under /home/user survives freezes, stops, restarts, and archiving. Learn what persists, what doesn't, and how rebuilds work.
---

Every sandbox owns a disk: a single large file on the host that acts as
a complete private disk (`DORMICE_SANDBOX_DISK_GB`, default 10 GB),
appearing inside the sandbox as `/home/user`. The
[container](/docs/core-concepts#containers-and-images) around it is
replaceable machinery. That gives you one rule to remember:

> **What lives in `/home/user` survives everything short of destroying
> the sandbox. Everything outside it is container state, and containers
> may be replaced at any time.**

See [Anatomy of a sandbox](/docs/architecture#anatomy-of-a-sandbox) for
how the disk, the container, and the image layers stack.

## What survives

All of these are exercised by the test suite against a real Docker +
gVisor host:

- **Freeze / stop / wake cycles** — files ride through every lifecycle
  transition.
- **Daemon restarts and host reboots** — the daemon checks its records
  against reality at boot, and the next wake re-attaches the disk.
- **`docker container prune`** (a Docker cleanup command that deletes
  stopped containers) removing the container — the next acquire builds
  a fresh container from the surviving disk: same sandbox ID,
  `/home/user` intact.
- **Out-of-memory** — a sandbox that hits its memory cap
  (`DORMICE_SANDBOX_MEMORY_GB`, default 2 GB) is killed with exit
  code 137, the Linux convention for "forcibly killed", and recorded as
  stopped. The same key acquires it right back, files intact.
- **Archive → restore** — the disk round-trips through S3 and comes
  back byte-for-byte.

## What does not

- **`destroySandbox`** — the only verb that loses data, by design.
- **Everything outside `/home/user`**, whenever the container is
  replaced (rebuild, prune, OOM): `/tmp`, packages installed at runtime
  outside the home directory, running processes. If it matters, keep it
  in `/home/user`.

## Rebuild a sandbox

You can replace a sandbox's container while keeping its disk — with the
SDK (`rebuildSandbox`), the CLI, or the console's Rebuild button:

```sh
dor sandbox rebuild my-agent
```

The next wake builds a fresh container from the daemon's *current* base
image (or the sandbox's template's current image), with `/home/user`
carried over untouched.

This is the upgrade path: update the image, re-register the
[template](/docs/templates), rebuild the sandbox — its files never move.
Rebuild does not wake the sandbox afterwards; it starts cold on its next
use.

## Restores upgrade too

When an [archived](/docs/archiving) sandbox is restored, Dormice builds
a **new** disk at the current `DORMICE_SANDBOX_DISK_GB` setting, unpacks
into it, and boots from the current image. Grow the disk setting while a
sandbox sits in the bucket and it comes back bigger — for free.

## How the disk quota works

The quota is physical: the disk image file is simply that big. A
sandbox that fills its disk gets `ENOSPC` ("no space left on device")
from its own filesystem — there is no accounting layer to game, and the
host's disk stays healthy. The disk file is sparse — the operating
system only stores the parts that have actually been written — so a
10 GB promise occupies only what the sandbox really uses. This means
the promises can add up to more than the host's disk (overcommit), and
[`getHostMetrics`](/docs/metrics) reports both the promised and the
real total so you can watch it.
