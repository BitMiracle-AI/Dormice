This page is the deep end: how Dormice works underneath, and the
security properties you are relying on. You don't need it to use
Dormice — start with the [Quick start](/docs/quickstart) — but read it
before you run untrusted code in production.

## One daemon, one database

The daemon keeps a SQLite database of sandboxes and drives Docker to
match it. **SQLite is the ledger, Docker is reality, and the daemon's
core job is keeping the two in agreement.**

- Reality moves first, then the ledger records it. A crash between the
  two leaves a container without a record — never a record without a
  container — and the reconciler cleans that up.
- A **reconciler** runs before the daemon starts listening and again on
  every heartbeat: a container that died behind the daemon's back is
  recorded as stopped (its disk, and therefore its data, kept), and
  ownerless leftovers are swept.
- The design is **crash-only**: there is no graceful-shutdown path to
  get wrong. You can kill the daemon at any moment; it reconciles on the
  way up.
- **One machine, one daemon**, enforced rather than assumed: an
  exclusive lock next to the database makes a second daemon refuse to
  start, and a startup guard refuses to boot when the database and the
  machine's reality cannot belong together (an empty database facing
  live sandboxes means a wrong database path — starting anyway would
  destroy them all).

There is deliberately no cluster mode, no external database, no queue.
The schema carries the fields sharding would need (node IDs, endpoints),
but today the simplicity is the feature.

## What a sandbox is

Each sandbox is a Docker container running under **gVisor** (`runsc`),
with its `/home/user` on a dedicated disk image (see
[Sandbox persistence](/docs/persistence)). Every sandbox runs with the
same non-negotiable parameters: non-root (uid 1000), `--init`,
`no-new-privileges`, a pids limit, CPU and memory caps.

Why gVisor and not VMs: freezing requires sandboxes to be **processes**
(suspend them, push their memory to swap), and "installs anywhere" rules
out requiring KVM. Firecracker-class hardware virtualization is a
different trade — see the last section.

## The security boundary

- **gVisor's userspace kernel is the boundary.** Sandbox code — even
  running as `root` inside — talks to gVisor's reimplemented kernel, not
  the host's. `uname -r` inside a sandbox reports a `-gvisor` kernel;
  the host kernel is never exposed.
- **Four rules hold everywhere:**
  1. The daemon binds to 127.0.0.1 only — by design, without a setting
     to change it. Exposing anything is a reverse proxy's explicit job.
  2. The Docker socket never enters a container.
  3. Containers cannot reach the cloud metadata service (blocked by
     iptables) or each other (`icc: false`).
  4. Containers may vanish at any time; only the disk is the sandbox.
- **Credentials never downgrade.** The API token must be 32+ characters
  and is compared in constant time. The E2B surface derives per-sandbox
  access tokens from it (HMAC, stateless). The web console trades it
  once for an httpOnly session cookie and never stores it anywhere page
  JavaScript can read.
- **Sandbox preview URLs are deliberately unauthenticated** (the
  `getHost()` port proxy, when `DORMICE_SANDBOX_DOMAIN` is set) — same
  as E2B: they exist to be opened by humans and shared. Don't run a
  service on a sandbox port unless you mean for the URL to work.

## When Dormice is the wrong tool

Pick something else if:

- **You need a fleet.** One machine, one daemon, by design — that is
  where the simplicity comes from. Multi-machine sharding is a future
  direction (the schema already carries the fields), not a current
  feature.
- **You want a managed service.** No hosted anything, no SLA. That is
  E2B's product, and it is good at it.
- **Your threat model demands hardware virtualization.** Sandboxes are
  Docker + gVisor — a userspace kernel, chosen deliberately, as above.
  If only Firecracker-class VM isolation will do, this trade is not for
  you.
- **Your workloads aren't Linux.** Sandboxes are Linux containers.
