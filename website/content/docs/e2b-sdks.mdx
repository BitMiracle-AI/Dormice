---
title: Using the official E2B SDKs
description: The official e2b packages — JavaScript and Python, unmodified — run against Dormice by changing two URLs.
---

The daemon speaks the E2B protocol on two prefixes (`/e2b/api` for the
control plane, `/e2b/envd` for the in-sandbox surface). Point the official
SDK at them and prefix your token with `e2b_` — migrating an application
is configuration, not code. Everything documented here is exercised by
black-box test suites *through the official packages* (e2b 2.31.0,
JavaScript and Python) against a real Docker + gVisor daemon.

## JavaScript

```ts
import { Sandbox } from 'e2b';

const sbx = await Sandbox.create({
  apiKey: `e2b_${process.env.DORMICE_API_TOKEN}`,
  apiUrl: 'http://127.0.0.1:3676/e2b/api',
  sandboxUrl: 'http://127.0.0.1:3676/e2b/envd',
});

await sbx.commands.run('echo hello');
```

## Python

Both the sync and the async client pass the suite:

```python
import os
from e2b import Sandbox

sandbox = Sandbox.create(
    api_key=f"e2b_{os.environ['DORMICE_API_TOKEN']}",
    api_url="http://127.0.0.1:3676/e2b/api",
    sandbox_url="http://127.0.0.1:3676/e2b/envd",
)

sandbox.commands.run("echo hello")
```

One constraint worth knowing: **keep the API token hexadecimal**
(`openssl rand -hex 32`, which is also what `install.sh` generates). The
Python SDK validates its key against `e2b_[0-9a-f]+` client-side, before
any request leaves the machine — a token with other characters fails in
the SDK, not on the wire.

## `metadata.userKey`: idempotent create

A Dormice extension: pass `metadata: { userKey: '...' }` and
`Sandbox.create` behaves like the native `acquireSandbox` — the same key
always returns the same sandbox instead of a new one, whatever state it
was in, with its files intact:

```ts
const sbx = await Sandbox.create({
  metadata: { userKey: 'my-project' },
  // ...the two URLs and apiKey as above
});
```

Without it, `create` makes a fresh sandbox every time — faithful E2B
semantics.

## Timeouts are real

`timeoutMs` and `setTimeout()` set an absolute deadline, exactly as E2B
defines them: at the deadline the sandbox is killed, or — with
`lifecycle: { onTimeout: 'pause' }` — parked, to be revived by `connect`.
See [Sandbox lifecycle](/docs/lifecycle) for how deadlines interact with
Dormice's own idle policy.

## Version gating

The daemon reports envd `0.6.1`, chosen so the SDK itself gates features
client-side: everything at or below that version is implemented, and the
few gaps above it are answered with an honest `unimplemented` on the wire
— the full list is in [Capabilities & differences](/docs/e2b-differences).
