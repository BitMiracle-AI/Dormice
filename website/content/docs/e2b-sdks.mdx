title: Using the official E2B SDKs
description: Learn how to run the official e2b packages — JavaScript and Python, unmodified — against Dormice by changing two URLs.
---

[E2B](https://e2b.dev) is a popular commercial sandbox cloud, and its
official `e2b` SDKs (JavaScript and Python) are what many agent
frameworks already build on ([what is
E2B?](/docs/core-concepts#what-is-e2b)). Dormice speaks the E2B
protocol, so those SDKs work against it without modification.
Migrating an application is configuration, not code: point the SDK at
your daemon's two URL prefixes and prefix your API token with `e2b_`.

If you have never used E2B, you don't need this page — the [native
API](/docs/quickstart) covers everything.

Everything documented here is exercised by black-box test suites
*through the official packages* (e2b 2.31.0, JavaScript and Python)
against a real Docker + gVisor daemon.

## JavaScript

```ts
import { Sandbox } from 'e2b';

const sbx = await Sandbox.create({
  apiKey: `e2b_${process.env.DORMICE_API_TOKEN}`,
  apiUrl: 'http://127.0.0.1:3676/e2b/api',
  sandboxUrl: 'http://127.0.0.1:3676/e2b/envd',
});

await sbx.commands.run('echo hello');
```

## Python

Both the sync and the async client pass the suite:

```python
import os
from e2b import Sandbox

sandbox = Sandbox.create(
    api_key=f"e2b_{os.environ['DORMICE_API_TOKEN']}",
    api_url="http://127.0.0.1:3676/e2b/api",
    sandbox_url="http://127.0.0.1:3676/e2b/envd",
)

sandbox.commands.run("echo hello")
```

**Note: keep the API token hexadecimal** (`openssl rand -hex 32`, which
is what `install.sh` generates). The Python SDK validates its key
against `e2b_[0-9a-f]+` on the client side, before any request leaves
the machine — a token with other characters fails in the SDK, not on the
wire.

[API keys](/docs/http-api#api-keys) minted through the daemon work here
too — `e2b_<key>` anywhere the token goes, and they are pure hex by
construction, so the Python SDK's validation never bites.

## Reuse the same sandbox across creates

A Dormice extension: pass `metadata: { externalId: '...' }` and
`Sandbox.create` behaves like the native `acquireSandbox` — the same key
always returns the same sandbox instead of a new one, whatever state it
was in, files intact:

```ts
const sbx = await Sandbox.create({
  metadata: { externalId: 'my-project' },
  // ...the two URLs and apiKey as above
});
```

Without it, `create` makes a fresh sandbox every time — faithful E2B
semantics.

## Timeouts work as E2B defines them

`timeoutMs` and `setTimeout()` set an absolute deadline: at the deadline
the sandbox is killed, or — with `lifecycle: { onTimeout: 'pause' }` —
paused, to be revived by `connect`. See
[Sandbox lifecycle](/docs/lifecycle) for how deadlines interact with
Dormice's own idle policy.

## What's implemented

The daemon reports envd `0.6.1`, chosen so the SDK itself gates features
on the client side: everything at or below that version is implemented,
and the few gaps above it are answered with an honest `unimplemented` on
the wire. The full list is in
[Capabilities & differences](/docs/e2b-differences).
